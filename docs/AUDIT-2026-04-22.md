# Comprehensive audit — lamboot-tools v0.2.0 rehearsal state

**Audit date:** 2026-04-22
**Audit scope:** entire LamBoot ecosystem (`lamboot-tools-dev`, `lamboot-dev`, `lamco-admin`, missing companion repos) against claimed v0.2.0 readiness.
**Audit method:** three parallel research-agents (internal claims, cross-repo dependencies, test-reality) + direct spade-work verification.
**Commissioning prompt:** founder's observation that Sessions A–Q were executed mechanically and may contain gaps, stubs, inconsistencies, or silent dependency assumptions.

---

## 0. Executive summary — the honest state

**`v0.2.0` is not ready to ship.** The `release-rehearsal.sh` 28/28 green is real, but it measures structural completeness (file presence, section headers, registry calls), not substance. Under substantive scrutiny, the release has **9 BLOCKER-class issues**, **9 GAP-class issues**, and **4 COSMETIC issues**. Critically, **three "stable" tools contain stubs**, the flagship `lamboot-migrate to-uefi` has **zero end-to-end tests**, the claimed cross-repo dependency (`lamboot-dev v0.8.4`) is **paused, not shipped**, and the PVE companion release path is **architecturally incomplete** (target repo doesn't exist; mirror scripts have never run).

The pattern underneath all findings: **the sessions optimized for "checkmark green" rather than substance.** `verify-claims.sh` checks that functions *exist*, not that they *work*. `release-rehearsal.sh` checks that files *are present*, not that their contents match their claims. The CHANGELOG makes count-based assertions ("8 actions", "10 subcommands", "25 pages") that are wrong by one in multiple places — the sign of uncounted claims rather than verified claims. The fleet-test matrix is documented for 26 VMs but has **zero baseline runs**. The specs and code have drifted.

**None of this is catastrophic.** Every blocker is fixable. The tool implementations are genuinely substantial — the code is there, the architecture holds up, the philosophy (diagnose → plan → show → confirm → execute → verify) is visible in every tool. The gap is between what's built and what's claimed, not between what's claimed and what's necessary. A focused remediation pass can honestly deliver v0.2.0 without ambition reduction.

**Recommended posture:** treat `v0.2.0` as "release-candidate-1" rather than release-ready. Do a remediation session (call it Session R) that addresses blockers, tightens claims to match reality, and produces honest test coverage evidence. Re-run rehearsal with substantive checks added. Then ship.

---

## 1. Severity key

- **BLOCKER** — violates a specific claim in CHANGELOG/ROADMAP/specs; users following docs will hit a wall; or release-engineering step will fail. Must fix before v0.2.0 ships.
- **GAP** — claim is technically true but overstated, or spirit-of-vision violation: silent flag acceptance, test that only asserts `--help` works, feature "deferred" without disclosure. Should fix before v0.2.0 ships; defer to v0.2.1 only with explicit CHANGELOG disclaimer.
- **COSMETIC** — count is off by one, phrasing is ambiguous, no user-observable wrongness. Fix opportunistically.

## 2. BLOCKER findings (9)

### B1 — Three "stable" tools contain unimplemented features

| Tool | Stability claim | Stub evidence | Effect on user |
|---|---|---|---|
| `lamboot-backup` v0.2.0 stable | CHANGELOG:24 | `tools/lamboot-backup:183-190` errors `"Offline NVRAM read not yet implemented"` while the registry at line 39 advertises `--offline-capable "true"` | User runs `lamboot-backup save --offline DISK`, gets a runtime error, questions whether ANY offline claim is real |
| `lamboot-migrate` v1.0.0 stable | CHANGELOG:26 | `tools/lamboot-migrate:845` warns `"--remove-grub specified but not implemented in v1.0 (manual removal required)"` | User runs `lamboot-migrate to-lamboot --remove-grub`, sees a warning, thinks GRUB was removed, later discovers it wasn't |
| `lamboot-doctor` v0.2.0 beta | CHANGELOG:27 | `tools/lamboot-doctor:593` silently accepts `--offline`; registry at line 55 declares `--offline-capable "false"`. **Worse than Agent A found:** the registry claims unsupported, code accepts silently — both outputs are wrong | User passes `--offline` (because it's a toolkit-wide convention), doctor does nothing with it and proceeds in online mode without warning |

**Decision needed per tool:**
- Either (a) downgrade stability label in CHANGELOG (`backup: stable → beta for v0.2`; `doctor: beta stays but registry + code aligned); or

**Recommendation:** (c) for all three — least code change, truthful outcome. Backup: drop `--offline-capable "true"` from registry, make `--offline` a `die` with `EXIT_PREREQUISITE`. ## 5. Cross-cutting patterns

### 5.1 The "checkmark-green" anti-pattern

The release-rehearsal infrastructure (28/28 green, verify-claims 24/24, CI green, 250 tests passing) presents as objective validation. Under scrutiny, it's mostly **structural** — checking that the shape of the release is right, not that the substance is right. This isn't malice; it's the natural drift of session-by-session execution optimizing for "this session's work landed." The fix isn't to stop using these scripts — they catch real regressions. The fix is to **stop treating them as claims-validation** and add substantive checks alongside them.

### 5.2 The claim/code drift pattern

Multiple "count" claims are off by one in the direction of overstatement (signing-keys 10 vs 9, repair 8 vs 9, website 25 vs 24, migrate-in-ROADMAP 10 vs 5). The common root cause: claims written before final counts, never re-counted at rehearsal. A 30-minute pass that literally counts each claim against code would catch most of these.

### 5.3 The silent-accept pattern

Three tools (`migrate --remove-grub`, `doctor --offline`, `backup --offline`) accept flags without implementing them — some warn, some error, some are silent. The toolkit philosophy ("diagnose → plan → show → confirm") is about honesty with users; silently accepting a flag is dishonest even if the code path doesn't execute. **Unified fix**: adopt a convention that every unimplemented-but-parsed flag calls `die` with `EXIT_PREREQUISITE` and a message explaining which version will support it. This is 10–20 lines total across three tools.

### 5.4 The dependency-claim vs dependency-reality pattern

`lamboot-tools-dev` claims coordination with `lamboot-dev v0.8.4` in multiple places. `lamboot-dev` has paused v0.8.4. Neither repo knew the other's state during session execution because the coordination mechanism (`CROSS-REPO-STATUS.md`) is a document, not a machine-readable manifest. **Fix:** either (a) a `lamco-admin/pipelines/lamboot-tools/WORKFLOW.md` that both repos read (see G4); or (b) at minimum, add a `scripts/check-cross-repo-deps.sh` that both repos run during their rehearsals and that fails loud if upstream hasn't shipped.

### 5.5 The fixture vacuum

The entire offline-mode story, the entire integration-test story, the entire Tier 1 fleet-test story — all depend on fixture disk images that don't exist. The session work built the harness (catalog, regen scripts, download script, CI wiring) but not the data. This is fixable but needs a session of actual fixture-building work, not just more harness.
